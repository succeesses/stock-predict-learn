# iFinD 财报项目记录

## 背景

当前 JusticePlutus 已经接入 iFinD，但现阶段只是把 iFinD 当作原有综合分析链路的增强输入。

实际跑下来的感受是：

- 能提升分析依据
- 但产品层面的辨识度不够强
- 用户很难感知“为什么一定要用 iFinD”

因此后续更适合单独新开一个项目，把 iFinD 最有壁垒的数据能力做成独立产品，而不是继续埋在总分析里。

## 核心判断

iFinD 最有价值的，不是普通行情，不是新闻搜索，而是以下三类结构化能力：

### 1. 财报细项

典型字段：

- 营业总收入
- 归母净利润
- 扣非净利润
- 毛利率
- 净利率
- ROE
- 资产负债率
- 经营现金流
- 存货
- 应收类指标

价值：

- 能做“利润质量分析”
- 能识别表面增长和真实增长的差异
- 能做财务健康度判断

### 2. 一致预期 / 盈利预测

典型能力：

- 市场对未来净利润的预期
- 多期预测值
- 预测增长率
- 预期变化方向

价值：

- 能做“实际 vs 预期”的比较
- 能做“估值是否已经透支增长”的判断
- 能做预期差分析

### 3. 公告 / 财报事件链

典型能力：

- 年报 / 季报 / 业绩预告发布日期
- 公告标题
- 公告原文入口
- 财报相关事件触发

价值：

- 适合事件驱动推送
- 适合做“财报发布即分析”
- 适合做公告解释器，而不是单纯新闻搬运

## 为什么值得独立立项

如果只把 iFinD 作为原有分析增强：

- 用户感知弱
- 和普通新闻搜索 / 通用财务摘要差异不够明显
- 很难体现订阅价值

如果把它做成单独财报产品：

- 产品边界更清晰
- 能突出 iFinD 的独特性
- 更适合财报季的短周期高价值使用
- 即使 iFinD 权限失效，也能整体停用，不影响主产品

## 推荐产品形态

推荐主形态：

- `事件驱动的财报体检推送`

不推荐把它做成：

- 普通行情产品
- 泛新闻摘要
- 另一份模糊的买卖建议

### 形态 1：财报体检卡

目标：

- 用一条高密度输出说明“这份财报到底好不好”

典型结构：

- 财报结论：超预期 / 符合预期 / 低于预期
- 质量结论：利润质量、现金流质量、杠杆风险
- 核心异动：营收、归母、扣非、毛利率、ROE、存货、应收
- 预期对比：实际 vs 一致预期
- 估值结论：当前 PE/PB 是否仍合理
- 一句话总结：最值得关注的重点

### 形态 2：财报 / 公告事件推送

目标：

- 在事件发生时提醒，而不是每天重复推送

典型触发条件：

- 年报发布
- 一季报发布
- 半年报发布
- 三季报发布
- 业绩预告
- 业绩快报
- 重大财务相关公告

### 形态 3：预期差雷达

目标：

- 识别“市场原本预期”和“实际结果”之间的差异

典型输出：

- 预期上修 / 下修
- 实际超预期 / 不及预期
- 估值和增长是否错配

## 时间窗口判断

如果 iFinD 权限只剩一个月，最值得做的是财报季相关产品，而不是全年型产品。

原因：

- 财报披露是强事件驱动
- 财报季里，iFinD 的结构化数据优势最明显
- 一个月内可以高密度验证价值

尤其适合覆盖：

- 年报季
- 一季报季

## 推荐的最小版本

如果新项目要快速验证价值，推荐只做一个最小版本：

### 项目定义

输入：

- 自选股列表
- iFinD token

触发：

- 每天扫描一次是否有新财报 / 新业绩预告 / 新相关公告

输出：

- 只对发生新事件的股票推送“财报体检卡”

### 最小卡片应包含

- 股票名 + 事件类型 + 公告日期
- 营收 / 归母 / 扣非 / ROE / 现金流
- 实际 vs 预期
- 财报质量结论
- 估值结论
- 风险点
- 一句话结论

## 项目边界建议

为了保证无侵入、可失效，建议新项目遵守这些边界：

- 与 JusticePlutus 主项目解耦
- 有 iFinD 就运行
- 没有 iFinD 就整体关闭
- 不依赖原主项目的技术面 / 筹码 / 搜索主链路
- 把重点放在财报、预期、公告三件套

## 与主项目的关系

建议关系是：

- JusticePlutus：综合分析、交易视角、日常跟踪
- iFinD 财报项目：公司基本面、财报事件、预期差

两者互补，但不要强耦合。

## 下一步建议

新开项目时，优先做：

1. 财报事件扫描
2. 财报体检卡生成
3. 通知推送

暂时不要优先做：

1. 全量行业数据库
2. 大而全研究平台
3. 复杂前端页面

先把“财报来了，能不能第一时间给出一份有价值的解释”做透。
